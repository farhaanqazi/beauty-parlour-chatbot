# Questioning Reference

Guidelines for extracting useful project context from users.

## Principles

- Ask one question at a time when possible
- Lead with the most important question
- Follow up on vague answers — "What does success look like?" is better than accepting "it should work"
- Don't ask for information you can infer from context
- Stop questioning when you have enough to write a plan — not when you've covered everything

## The One-Sentence Test

Every project should reduce to: "[Name] helps [user] do [thing] by [mechanism]."
If you can't write this from the answers, ask more questions.

## Red Flags in Answers

- "It should be like [big product] but simpler" → Ask: what are the 3 core features, specifically?
- "We'll figure out the details later" → Fine for v2, but v1 needs scope. Push gently.
- "It needs to scale" → Scale to what? 10 users or 10 million? Changes everything.
- No mention of users → Who is this actually for?
